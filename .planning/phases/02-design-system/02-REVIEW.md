## Summary

The Phase 2 design system foundation is in solid shape: the prototype port is faithful, tests are present for all three custom primitives and all four format helpers, and the token layering in `globals.css` (prototype tokens → shadcn aliases → Tailwind 4 `@theme inline`) is well-commented and intentional. No security issues, no critical bugs, no commit-blocking concerns.

The `@theme inline` font-variable self-reference (W-03) is the most likely-to-bite item — worth a quick repro check in the dev server before Phase 3 starts depending on `font-display`.

The 9 shadcn-installed primitives in `src/components/ui/` are vendor code generated by `shadcn-CLI` (`base-vega` style); they are **not** custom-authored and were not deeply critiqued per scoping instructions. Spot-check confirmed they wire to the alias tokens in `globals.css` correctly. See the grouped note below.

## Warnings

### WR-01: TrustBadge sets `role="button"` on a `<span>` without keyboard support

**File:** `src/components/design/TrustBadge.tsx:25-33`
**Issue:** When `onClick` is provided the component sets `role="button"` on a `<span>`, but `<span>` is not focusable, no `tabIndex={0}` is set, and there is no `onKeyDown` handler for Enter/Space. Screen-reader users will hear "button" but cannot activate it; keyboard-only users cannot reach it at all. This is the same accessibility gap the prototype had (CONVENTIONS.md notes `:focus` is rare) — but Phase 2 is the moment to fix it before the badge gets reused on every KPI tile.
**Fix:**
```tsx
export function TrustBadge({ score, onClick, size = "sm" }: TrustBadgeProps): JSX.Element {
  const tier = tierFor(score);
  const interactive = Boolean(onClick);
  return (
    <span
      className={`trust ${tier}`}
      data-size={size}
      onClick={(e) => {
        e.stopPropagation();
        onClick?.();
      }}
      onKeyDown={
        interactive
          ? (e) => {
              if (e.key === "Enter" || e.key === " ") {
                e.preventDefault();
                e.stopPropagation();
                onClick?.();
              }
            }
          : undefined
      }
      role={interactive ? "button" : undefined}
      tabIndex={interactive ? 0 : undefined}
    >
      <span className="trust-dot" /> {score}%
    </span>
  );
}
```
Alternative: render as `<button type="button">` when `onClick` is set and reset native button styling in `globals.css`. Cleaner, but changes the DOM shape from the prototype.

### WR-02: TrustBadge accepts arbitrary `score` without validating range; tier formula treats `NaN` as "bad"

**File:** `src/components/design/TrustBadge.tsx:10-14, 34`
**Issue:** `tierFor(NaN)` returns `"bad"` (both `>= 90` and `>= 70` are false for NaN). The badge will then render `NaN%` in its label. Score values outside `[0, 100]` (e.g. a backend bug returning `150` or `-10`) render verbatim as `150%` / `-10%`. The product semantics in D-08 say "score is a percentage 0–100" — defensive clamping or a dev-time warning would catch upstream regressions early.
**Fix:**
```tsx
function tierFor(score: number): Tier {
  if (!Number.isFinite(score)) return "bad";
  if (score >= 90) return "good";
  if (score >= 70) return "warn";
  return "bad";
}

export function TrustBadge({ score, onClick, size = "sm" }: TrustBadgeProps): JSX.Element {
  const safeScore = Number.isFinite(score) ? Math.max(0, Math.min(100, Math.round(score))) : 0;
  const tier = tierFor(safeScore);
  // …render `safeScore`% instead of raw `score`
}
```

### WR-03: `@theme inline` self-references next/font CSS variables — likely to resolve correctly but worth verifying

**File:** `src/app/globals.css:77-80, 198-201`
**Issue:** Layer 1 defines `--f-sans: var(--font-sans)` and the `@theme inline` block defines `--font-sans: var(--font-sans)`. Both reference the same CSS-variable name that next/font sets on `<html className={inter.variable}>` in `layout.tsx`. With Tailwind 4 `@theme inline`, the right-hand side is preserved as a `var()` reference (rather than resolved at build time), so cascade order should make this work — but the self-named pattern (`--font-sans: var(--font-sans)`) is fragile: a future refactor that drops `inline` or changes the variable name on either end will silently fall back to the browser default font.
**Fix:** Either rename next/font variables to a non-colliding token (e.g. `variable: "--nextfont-sans"`) and chain explicitly, or document the dependency at the top of `globals.css` so future maintainers don't break it. A spot check in dev (`document.body` computed `font-family` should resolve to "Inter") on an empty page confirms the chain is wired today.
```ts
// layout.tsx — option A: rename to avoid collision
const inter = Inter({ /* … */ variable: "--nextfont-sans" });
```
```css
/* globals.css */
:root { --f-sans: var(--nextfont-sans); }
@theme inline { --font-sans: var(--nextfont-sans); }
```
If you keep the current names, add a header comment in `globals.css`:
```css
/* IMPORTANT: --font-sans / --font-display / --font-mono are set on <html> by next/font in layout.tsx.
   The @theme inline block re-declares them by the same name to expose them as Tailwind theme tokens.
   Do NOT remove `inline` — without it Tailwind 4 resolves the var() at build time and breaks the chain. */
```

### IN-01: `fmtCompact` rolls 999_500+ into "€1000K" instead of "€1M"

**File:** `src/lib/format/index.ts:25-33`
**Issue:** Boundary case: `fmtCompact(999_500)` → `Math.round(999.5) = 1000` → `"€1000K"`. The expected reading is `"€1,0M"` once the rounded thousands figure crosses 1000. Not a Phase 2 blocker — no current callsite hits this band — but worth tightening before charts/KPI tiles consume it.
**Fix:**
```ts
export function fmtCompact(n: number): string {
  if (!Number.isFinite(n)) return fmtEUR(0);
  const abs = Math.abs(n);
  if (abs >= 1_000_000 || Math.round(abs / 1_000) >= 1000) {
    return `€${(n / 1_000_000).toLocaleString("nl-NL", { maximumFractionDigits: 1 })}M`;
  }
  if (abs >= 1_000) {
    return `€${Math.round(n / 1_000)}K`;
  }
  return fmtEUR(n);
}
```

### IN-02: `fmtCompact` does not handle `NaN` / `Infinity`

**File:** `src/lib/format/index.ts:25-33`
**Issue:** `fmtCompact(NaN)` falls through to `fmtEUR(NaN)`, which returns `"€ NaN"` (literal). `fmtCompact(Infinity)` returns `"€Infinity..."`. The test suite covers happy paths only. Cheap to defend.

### IN-03: `AskButton` always calls `e.stopPropagation()` — even with no `onClick`

**File:** `src/components/design/AskButton.tsx:27-30`
**Issue:** `e.stopPropagation()` runs unconditionally, including when the component is decorative (no `onClick` prop). Phase 3 will use outside-click detection for drawers and dropdowns (CONVENTIONS.md describes the pattern); a stray AskButton in a drawer body that swallows the bubble could break that detection. Test `WR-AskButton.test.tsx` line 32 in fact codifies the current behavior, so this is a deliberate-but-questionable choice — flagging because the comment in the source ("Rule-3 fix vs prototype") explains the `type="button"` change but not the `stopPropagation` decision.
**Fix:** Only stop propagation when an `onClick` is wired:
```tsx
onClick={(e) => {
  if (onClick) {
    e.stopPropagation();
    onClick(e);
  }
}}
```
Update the test to reflect: parent receives the bubble when `onClick` is omitted; doesn't receive it when `onClick` is set.

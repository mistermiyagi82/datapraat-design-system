# Phase 1: Code Review Report

**Reviewed:** 2026-04-26
**Depth:** standard
**Files Reviewed:** 40
**Status:** issues_found

## Summary

Phase 1 foundation code is well-structured, documented, and follows the planning artifacts closely. The storage seam, migration runner, health route, env parsing, and Docker/Railway plumbing are all coherent and testable. No critical bugs or security issues were found.

The main concern is a **pnpm version mismatch between Nixpacks and the rest of the toolchain** (Dockerfile, package.json, lockfile all use pnpm 10; nixpacks.toml pins pnpm 9). This is the most likely source of a Railway build failure and should be fixed before deploy.

A second class of warnings concerns runtime fragility: `process.cwd()`-relative migration loading, a dev-boot script that doesn't verify the bound port, and an entrypoint chown that silently swallows failures. None are bugs in the happy path, but each removes a guard rail against a real failure mode.

Info-level findings cover stale create-next-app boilerplate (Geist font references), redundant data in `health_probe`, and minor type-assertion style.

## Warnings

### WR-01: pnpm version mismatch between Nixpacks and the rest of the toolchain

**File:** `nixpacks.toml:6`
**Issue:** `nixPkgs = ["nodejs_22", "pnpm-9_x", ...]` pins pnpm 9, but `package.json` declares `"packageManager": "pnpm@10.30.2"`, the `Dockerfile` activates `pnpm@10.30.2` via Corepack, and `pnpm-lock.yaml` was generated by pnpm 10. Running `pnpm install --frozen-lockfile` (line 9) under pnpm 9 against a v10 lockfile will either fail outright or silently produce a different `node_modules` than the Docker build, defeating the "same artifact everywhere" goal.
**Fix:**
```toml
[phases.setup]
nixPkgs = ["nodejs_22", "python3", "gcc", "gnumake"]

[phases.install]
cmds = [
  "corepack enable",
  "corepack prepare pnpm@10.30.2 --activate",
  "pnpm install --frozen-lockfile",
]
```
This delegates pnpm-version selection to Corepack (matching the Dockerfile pattern) and removes the drift risk.

### WR-02: Migration runner depends on `process.cwd()` matching the project root

**File:** `src/lib/storage/sqlite/migrate.ts:12`
**Issue:** `MIGRATIONS_DIR = path.join(process.cwd(), "src/lib/storage/sqlite/migrations")` works only when the server is launched with the project root as CWD. The Dockerfile copies migrations to `/app/src/lib/storage/sqlite/migrations` and the standalone server is started with `WORKDIR /app`, so this happens to work — but any consumer that runs `node /app/server.js` from another directory (some PaaS, some debug commands, future ECS/Container Apps task definitions) will silently load zero migrations and leave `health_probe` missing, producing 503s with no clear cause. The Dockerfile comment (line 52-54) already flags this as fragile.
**Fix:** Resolve the directory relative to the module file instead of the working directory:
```ts
import { fileURLToPath } from "node:url";

const MIGRATIONS_DIR = path.join(
  path.dirname(fileURLToPath(import.meta.url)),
  "migrations",
);
```
Then drop the explicit Dockerfile copy at line 54 in favour of letting Next.js standalone tracing follow the import (which it can, once the path is module-relative).

### WR-03: docker-entrypoint silently ignores chown failures

**File:** `docker-entrypoint.sh:9`
**Issue:** `chown -R nextjs:nodejs "$DATA_DIR" 2>/dev/null || true` swallows every chown failure, including the one this script exists to fix. If the volume is read-only, mounted with the wrong user namespace, or `$DATA_DIR` doesn't exist (the `if [ -d ... ]` covers only the last case), the app starts as `nextjs` and fails on the first sqlite write — producing a 503 from `/api/health` with the misleading "probe roundtrip" error rather than the actual permissions problem. Combined with `set -e` on line 2, the `|| true` is a deliberate decision; it should at least log.
**Fix:**
```sh
DATA_DIR="$(dirname "${DB_PATH:-/data/datapraat.sqlite}")"
if [ -d "$DATA_DIR" ]; then
  if ! chown -R nextjs:nodejs "$DATA_DIR" 2>/tmp/chown.err; then
    echo "[entrypoint] WARN: chown $DATA_DIR failed: $(cat /tmp/chown.err)" >&2
  fi
else
  echo "[entrypoint] WARN: $DATA_DIR does not exist; sqlite will fail on first write" >&2
fi
```

### WR-04: `check-dev-boot.sh` polls a hard-coded port that may not match the bound port

**File:** `scripts/check-dev-boot.sh:5,16`
**Issue:** `PORT="${PORT:-3000}"` then polls `http://localhost:${PORT}/`. If port 3000 is already in use on the developer's machine, Next.js 15 dev server falls back to 3001/3002/etc., and the check polls the wrong port for 30s and reports a false negative. The dev log tail at the end will reveal it, but the first failure mode is confusing.
**Fix:** After starting `pnpm dev`, parse the actual bound port from the dev log before polling:
```sh
# Wait briefly for Next.js to print the bound URL
for i in $(seq 1 30); do
  ACTUAL_PORT="$(grep -oE 'http://localhost:[0-9]+' /tmp/datapraat-dev.log | head -1 | grep -oE '[0-9]+$' || true)"
  if [ -n "$ACTUAL_PORT" ]; then break; fi
  sleep 1
done
PORT="${ACTUAL_PORT:-$PORT}"
```
Or, simpler: pass `-p "$PORT"` to `pnpm dev` so Next.js binds to exactly that port (and fails loudly if it can't).

## Info

### IN-01: `globals.css` references Geist fonts that aren't loaded

**File:** `src/app/globals.css:11-12`
**Issue:** `--font-sans: var(--font-geist-sans);` and `--font-mono: var(--font-geist-mono);` reference CSS variables that `next/font/google` would inject — but `src/app/layout.tsx` does not load any next/font, and the body uses `font-family: Arial, Helvetica, sans-serif` (line 25). These Geist references are stale boilerplate from `create-next-app` and resolve to `unset`, falling back to Arial. Either remove them or load Geist properly in `layout.tsx`.
**Fix:** Remove lines 11-12 of `globals.css` and the dark-mode block at lines 15-20 if not used in Phase 1, OR replace with the project's actual fonts (Inter/Fraunces/JetBrains Mono per `CLAUDE.md`).

### IN-02: `health_probe` stores the same value in two columns

**File:** `src/lib/storage/sqlite/health-probe.ts:18` and `migrations/0001_init.sql:5-9`
**Issue:** `recordProbe(key, atMs)` runs `INSERT INTO health_probe (key, value, updated_at) VALUES (?, ?, ?)` with `String(atMs)` in `value` and `atMs` in `updated_at`. The `value` column is never read (`readProbe` returns `updated_at`). Either delete the `value` column or use it for something distinct (e.g., a probe payload).
**Fix:** Drop `value` from the schema (and the insert) — this also future-proofs the table for keys whose payload genuinely differs from the timestamp.

### IN-03: `tsconfig.json` excludes test files from typecheck

**File:** `tsconfig.json:33-35`
**Issue:** `**/*.test.ts`, `**/*.test.tsx`, and `scripts/**/*.test.ts` are excluded from the TS program, so `pnpm typecheck` does not catch TS errors in tests. Vitest happens to transpile + run them, but type errors in test code only surface at test time, not in CI typecheck. For Phase 1 this is fine; mention here so it doesn't surprise later phases.
**Fix:** Optional — add a separate `tsconfig.test.json` that includes tests and run `tsc -p tsconfig.test.json --noEmit` in CI.

### IN-04: Type-assertion style in health route widens literal types unnecessarily

**File:** `src/app/api/health/route.ts:20-26`
**Issue:** `status: "ok" as "ok" | "degraded"` and `db: { ok: true as boolean, probeMs: 0 as number }` use `as` to broaden a value's inferred type. A typed `let result: HealthBody = { ... }` is cleaner and gives the same widening for free.
**Fix:**
```ts
type HealthBody = {
  status: "ok" | "degraded";
  commitSha: string;
  buildTime: string;
  nodeEnv: string;
  uptimeSec: number;
  db: { ok: boolean; probeMs: number };
};

const result: HealthBody = {
  status: "ok",
  commitSha: process.env.NEXT_PUBLIC_COMMIT_SHA ?? "unknown",
  buildTime: process.env.BUILD_TIME ?? "unknown",
  nodeEnv: env.NODE_ENV,
  uptimeSec: Math.round(process.uptime()),
  db: { ok: true, probeMs: 0 },
};
```

### IN-05: `env.ts` reads raw `process.env.NODE_ENV` inside its own schema default

**File:** `src/lib/env.ts:19`
**Issue:** The `DB_PATH` default uses `process.env.NODE_ENV === "production"` directly, before Zod has parsed `NODE_ENV`. This works (Zod's default for `NODE_ENV` is also `"development"`, matching the raw fallback), but the parallel sources of truth are easy to skew. If the `NODE_ENV` enum or default changes, `DB_PATH`'s default could silently disagree.
**Fix:** Compute the path-default lazily after parsing, or use `.transform` / `.refine` so the default depends on the parsed object rather than raw `process.env`. Simplest:
```ts
const RawSchema = z.object({
  NODE_ENV: z.enum(["development", "test", "production"]).default("development"),
  // ... other fields, but with DB_PATH still optional
  DB_PATH: z.string().min(1).optional(),
  // ...
});

function parseEnv() {
  const raw = RawSchema.safeParse(process.env);
  if (!raw.success) { /* ... */ }
  const data = raw.data;
  return {
    ...data,
    DB_PATH:
      data.DB_PATH ??
      (data.NODE_ENV === "production" ? "/data/datapraat.sqlite" : "./.data/datapraat.sqlite"),
  };
}
```

### IN-06: `.dockerignore` excludes `**/*.test.ts` but not `**/*.test.tsx`

**File:** `.dockerignore:57`
**Issue:** Tests are excluded with `**/*.test.ts` only. Once Phase 2+ adds React component tests as `*.test.tsx`, they will be copied into the image (mostly harmless because standalone tracing won't include them, but they pollute build context and slow `COPY . .`).
**Fix:** Add `**/*.test.tsx` next to the existing `**/*.test.ts` line.
